<task type="checkpoint:decision" gate="blocking">
  <decision>Which hosting platform to deploy on?</decision>
  <context>
    The app is a Node.js/Express server (not static-only). It serves the Vite-built React SPA in production
    (NODE_ENV=production → serves dist/ as static files) plus Express routes for Google OAuth, Calendar API,
    and email invitations via Resend. It needs: a persistent process, custom domain support, environment
    variable management, and must work with Namecheap DNS CNAME/A records.
  </context>
  <options>
    <option id="fly-io">
      <name>Fly.io</name>
      <pros>
        - Docker-based deploy (full control over runtime)
        - Free tier: 3 shared-CPU VMs, 256MB RAM each
        - Custom domain + TLS certificate automated via flyctl
        - Namecheap: point CNAME to &lt;app&gt;.fly.dev
        - CLI-driven: flyctl deploy
        - Persistent processes (no cold starts)
      </pros>
      <cons>
        - Requires writing a Dockerfile
        - CLI setup (flyctl install + flyctl auth)
        - Slightly more ops overhead than Railway
      </cons>
    </option>
    <option id="railway">
      <name>Railway</name>
      <pros>
        - Zero-config Node.js detect (no Dockerfile needed)
        - GitHub-connected, auto-deploy on push
        - Custom domain with one-click TLS
        - Namecheap: CNAME to Railway-provided domain
        - $5 free credit/month (covers low-traffic MVP)
      </pros>
      <cons>
        - Free tier has usage limits (sleeps after inactivity on hobby plan)
        - Less control over runtime environment
        - Paid plan required for always-on ($5/month Hobby)
      </cons>
    </option>
    <option id="render">
      <name>Render</name>
      <pros>
        - Free tier for web services
        - Zero-config Node.js support
        - Custom domain + TLS
        - GitHub auto-deploy
      </pros>
      <cons>
        - Free tier has 15-min cold start spin-down (bad UX for a real app)
        - Paid plan ($7/month) required to avoid cold starts
      </cons>
    </option>
  </options>
  <resume-signal>Select: fly-io, railway, or render — or describe another platform</resume-signal>
</task>

<task type="auto">
  <name>Task 1: Fix PORT and create deployment config</name>
  <files>app/server.ts, Dockerfile, .dockerignore, fly.toml OR railway.toml (based on decision)</files>
  <action>
    **1. Fix hardcoded PORT in app/server.ts:**
    Change `const PORT = 3000;` to `const PORT = Number(process.env.PORT) || 3000;`
    This is required by virtually all hosting platforms that inject PORT at runtime.

    **2a. If fly-io was selected — create Dockerfile at repo root:**
    ```
    FROM node:20-alpine AS builder
    WORKDIR /app
    COPY app/package*.json ./
    RUN npm ci
    COPY app/ .
    RUN npm run build

    FROM node:20-alpine
    WORKDIR /app
    COPY --from=builder /app/package*.json ./
    RUN npm ci --omit=dev
    COPY --from=builder /app/dist ./dist
    COPY --from=builder /app/server.ts ./
    # server.ts uses tsx — include it
    RUN npm install tsx
    EXPOSE 8080
    ENV PORT=8080
    ENV NODE_ENV=production
    CMD ["npx", "tsx", "server.ts"]
    ```

    Also create fly.toml at repo root:
    ```toml
    app = "gradevinski-dnevnik"
    primary_region = "fra"

    [build]

    [http_service]
      internal_port = 8080
      force_https = true
      auto_stop_machines = "stop"
      auto_start_machines = true
      min_machines_running = 0
    ```

    Create .dockerignore at repo root:
    ```
    node_modules
    .paul
    docs
    *.md
    .git
    app/.env
    app/.env.*
    !app/.env.example
    ```

    **2b. If railway was selected — create railway.toml at repo root:**
    ```toml
    [build]
    buildCommand = "cd app && npm ci && npm run build"

    [deploy]
    startCommand = "cd app && node --loader tsx/esm server.ts"
    restartPolicyType = "ON_FAILURE"
    restartPolicyMaxRetries = 3
    ```
    Note: Railway auto-detects Node, but explicit config avoids surprises.

    **2c. If render was selected — create render.yaml at repo root:**
    ```yaml
    services:
      - type: web
        name: gradevinski-dnevnik
        runtime: node
        buildCommand: cd app && npm ci && npm run build
        startCommand: cd app && npx tsx server.ts
        envVars:
          - key: NODE_ENV
            value: production
    ```

    Avoid: committing any actual .env file. Config files only.
  </action>
  <verify>Run `npm run build` in app/ — confirm it exits 0 with no errors. Confirm config file exists at repo root.</verify>
  <done>AC-4 satisfied: PORT env var respected. Deployment config committed.</done>
</task>

<task type="checkpoint:human-verify" gate="blocking">
  <what-built>
    Deployment config + PORT fix are committed. Now deploy to the chosen platform and configure DNS.
  </what-built>
  <how-to-verify>
    **Step 1 — Deploy to platform:**

    *If Fly.io:*
    1. Install flyctl: `https://fly.io/docs/hands-on/install-flyctl/`
    2. `flyctl auth login`
    3. `flyctl launch --no-deploy` (first time — creates app, edit fly.toml if needed)
    4. Set secrets: `flyctl secrets set VITE_SUPABASE_URL=... VITE_SUPABASE_PUBLISHABLE_KEY=... APP_URL=... GOOGLE_CLIENT_ID=... GOOGLE_CLIENT_SECRET=... GEMINI_API_KEY=... NODE_ENV=production`
    5. `flyctl deploy`
    6. Note the assigned URL (e.g. `gradevinski-dnevnik.fly.dev`)

    *If Railway:*
    1. Go to railway.app → New Project → Deploy from GitHub repo
    2. Add all env vars in the Railway dashboard (Settings → Variables)
    3. Note the assigned Railway domain

    *If Render:*
    1. Go to render.com → New Web Service → Connect GitHub repo
    2. Add all env vars in Render dashboard (Environment tab)
    3. Note the assigned Render URL

    **Step 2 — Configure DNS at Namecheap:**
    1. Login to Namecheap → Domain List → gradevinskidnevnik.online → Manage → Advanced DNS
    2. Add a CNAME record:
       - Host: `elektro`
       - Value: `<your-platform-domain>` (e.g. gradevinski-dnevnik.fly.dev)
       - TTL: Automatic
    3. Wait for DNS propagation (usually 5–30 min, up to 24h)

    **Step 3 — Add custom domain on platform:**
    - Fly.io: `flyctl certs add elektro.gradevinskidnevnik.online`
    - Railway: Dashboard → Settings → Domains → Add custom domain
    - Render: Dashboard → Settings → Custom Domains → Add

    **Step 4 — Update Google OAuth authorized redirect URIs:**
    1. Go to Google Cloud Console → APIs & Services → Credentials → OAuth 2.0 Client
    2. Add: `https://elektro.gradevinskidnevnik.online/auth/google/callback`
    3. Save

    **Step 5 — Smoke test:**
    - [ ] https://elektro.gradevinskidnevnik.online loads landing page
    - [ ] HTTPS lock icon present
    - [ ] "Prijavi se" / "Registriraj se" CTA navigates to signup
    - [ ] Supabase login completes (Google sign-in popup)
    - [ ] App dashboard renders after login
    - [ ] No console errors about missing env vars
  </how-to-verify>
  <resume-signal>Type "deployed" when live and smoke tests pass, or describe any issues</resume-signal>

<objective>
## Goal
Detect discipline from URL subdomain or query param, show discipline-specific login/onboarding UI, and pre-set discipline when creating a new organization.

## Purpose
Completes the vertical expansion — discipline context flows from first touch (the landing URL) through org creation, not just after login. A contractor who lands on `voda.gradevinski.hr` gets water-specific terminology from the first screen and their org is automatically classified as water.

## Output
- `detectDisciplineFromSubdomain()` utility + `DISCIPLINE_SUBTITLES` map in `disciplineConfig.ts`
- `createOrganizationWithOwner` extended with optional `discipline` param in `data.ts`
- Login screen heading/description adapts to detected discipline in `App.tsx`
- Onboarding shows discipline selector pre-filled with detected discipline; passes it on org creation
</objective>

<task type="auto">
  <name>Task 1: Subdomain detection utility + data layer extension</name>
  <files>app/src/lib/disciplineConfig.ts, app/src/lib/data.ts</files>
  <action>
    **disciplineConfig.ts — add after existing exports:**

    1. Add `SUBDOMAIN_DISCIPLINE_MAP` — maps URL subdomain prefix to Discipline:
       ```
       elektro → 'electro'
       voda    → 'water'
       klima   → 'klima'
       ```

    2. Add `DISCIPLINE_SUBTITLES: Record<Discipline, string>` for login screen copy:
       ```
       electro: 'Profesionalni dnevni izvještaji za električare u manje od 2 minute.'
       water:   'Profesionalni dnevni izvještaji za vodoinstalatera u manje od 2 minute.'
       klima:   'Profesionalni dnevni izvještaji za klimatičare u manje od 2 minute.'
       ```

    3. Add `detectDisciplineFromSubdomain(): Discipline` function:
       - Read `window.location.hostname`, split on '.', take first segment, lowercase
       - If it matches a key in SUBDOMAIN_DISCIPLINE_MAP, return the mapped Discipline
       - Else check `new URLSearchParams(window.location.search).get('discipline')`
       - If the param value is a valid Discipline key (in DISCIPLINE_PHASES), return it cast as Discipline
       - Otherwise return 'electro'

    **data.ts — extend createOrganizationWithOwner:**

    4. Add `discipline?: Discipline` to the params object type (import Discipline from disciplineConfig if needed — check if already imported; if not, add the import)
    5. In the `.insert({...})` call for organizations, spread `discipline: params.discipline ?? 'electro'` so it's always set explicitly (not relying on DB default)

    Avoid: importing from disciplineConfig inside data.ts if it creates a circular dep — instead inline the type as `'electro' | 'water' | 'klima'` for the param, matching the existing discipline field type already in the Company interface.
  </action>
  <verify>Run `npm run build` (or `npx tsc --noEmit`) from the app directory — zero TypeScript errors. Check that detectDisciplineFromSubdomain is exported and DISCIPLINE_SUBTITLES has all 3 keys.</verify>

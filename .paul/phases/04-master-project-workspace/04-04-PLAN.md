  <name>Task 1: Create master_project_issues migration</name>
  <files>supabase/migrations/20260328_09_master_issues.sql</files>
  <action>
    Create new migration file with:

    Table `public.master_project_issues`:
    - id uuid primary key default gen_random_uuid()
    - master_project_id uuid references public.master_projects(id) on delete cascade
    - title text not null
    - description text
    - status text not null default 'open' check (status in ('open', 'in_progress', 'resolved', 'closed'))
    - priority text not null default 'medium' check (priority in ('low', 'medium', 'high'))
    - discipline text check (discipline in ('electro', 'water', 'klima'))  — nullable, issue may span disciplines
    - reported_by uuid references auth.users(id)
    - created_at timestamptz default now()

    Index: master_project_issues_master_idx on (master_project_id)

    RLS (enable + 3 policies):
    - SELECT: auth.uid() is not null AND (owner org OR participant org) — same pattern as master_project_organizations_select
    - INSERT: auth.uid() is not null AND master_project_id in (select from master_projects where owner OR master_project_id in participant rows) — all participants can report issues
    - UPDATE: auth.uid() is not null AND (reported_by = auth.uid() OR master_project_id in owner org) — reporter or owner can update
    - DELETE: auth.uid() is not null AND master_project_id in owner org only

    SELECT and INSERT policies must allow all participant orgs (not just owner).
    For INSERT, use the same two-branch OR pattern as master_project_organizations_select:
    owner_org OR (org is participant in master_project_organizations where org_id in user's org).

    Avoid recursive subqueries — follow the flat OR pattern from 20260328_07_master_workspace.sql.
  </action>

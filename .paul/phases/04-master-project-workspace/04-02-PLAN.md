## Goal
Add aggregated diary stats per discipline card and a cross-discipline activity feed to the MasterProjectDetailView — no new migrations needed.

## Purpose
Phase 4's core value is a unified status view across disciplines. Right now discipline cards show org name and role but no diary activity. This plan delivers the first layer of aggregation: entry counts, hours, and a shared activity timeline.

## Output
- 2 new exported functions + 2 interfaces in data.ts
- Discipline cards show entry count + total hours per linked project
- "Nedavna aktivnost" activity feed section in MasterProjectDetailView (10 most recent entries across all linked projects)

## AC-1: Discipline cards show aggregated stats
```gherkin
Given a master project with at least one linked org whose project has diary entries
When the user opens the master project detail view
Then each discipline card shows entry count and total logged hours for its linked project
```

## AC-2: Activity feed shows recent cross-discipline entries
```gherkin
Given a master project with linked organizations that have diary entries
When the user views the master project detail
Then a "Nedavna aktivnost" section displays the 10 most recent diary entries across all linked projects, each showing date, title, discipline icon, and status badge
```

## AC-3: Graceful empty and inaccessible states
```gherkin
Given a discipline card whose linked project has no diary entries OR whose entries are not accessible via RLS
When the user views the master project detail
Then the card shows "0 zapisa · 0h" without error, and the activity feed shows an empty state message
```

  <action>
    Add two exported interfaces near the Master Project Workspace section (line ~1062):

    ```ts
    export interface MasterProjectStats {
      projectId: string;
      entryCount: number;
      totalHours: number;
      lastEntryDate?: string;
    }

    export interface MasterActivityItem {
      entryId: string;
      projectId: string;
      entryDate: string;
      title: string;
      status: string;
      createdByName: string;
    }
    ```

    Add two exported async functions after `updateMasterProject`:

    `fetchMasterProjectStats(projectIds: string[]): Promise<MasterProjectStats[]>`
    - If projectIds is empty, return []
    - Query: `supabase.from('diary_entries').select('project_id, hours, entry_date').in('project_id', projectIds)`
    - Aggregate in JS: group rows by project_id, count rows, sum hours (treat null as 0), find max entry_date per group
    - Return array of MasterProjectStats, one per projectId that had rows

    `fetchMasterRecentActivity(projectIds: string[], limit = 10): Promise<MasterActivityItem[]>`
    - If projectIds is empty, return []
    - Query: `supabase.from('diary_entries').select('id, project_id, entry_date, title, status, created_by_name').in('project_id', projectIds).order('entry_date', { ascending: false }).limit(limit)`
    - Map rows to MasterActivityItem (id → entryId, created_by_name → createdByName)

    Note: RLS on diary_entries is scoped to organization_id — entries from other orgs' projects are only visible if the current user is a project_member. Stats and feed show only what RLS allows; AC-3 covers the zero case.
  </action>
